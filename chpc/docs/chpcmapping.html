Submitting a job to the CHPC

Today you will creating a script that can be used to submit a job to the CHPC. In this hypothetical example, you will discover “SNPs”. In reality, you will be mapping reads from E. Coli K-12 to the E Coli K-12 reference genome and detect positions with differing nucleotides. While this is obviously not “SNP” discovery in the strictest sense, the technical procedure to discover SNPs is very similar.

Most of you do not have CHPC accounts yet, so I will be submitting the jobs on your behalf. However, you will configure the script in such a way that the report on the job, i.e. on completion/error will be sent to your email address.

I should note that the scripts you will write are trivial and that advanced aspects of job submission will only be covered in other lectures.

Anatomy of a PBS script

The structure of a simple PBS script can be found here. You can reference it to determine what needs to be done. For those of you that are on the CHPC already, you may need to change the project number.

The Task

A researcher approached you with an interesting problem. Can you determine the nucleotide differences of the E. Coli they have been studying with the K-12 reference. The researcher has already sequenced the organism and has the reads available. You need to map these reads to the K-12 reference genome. However, you already have a lot of tasks to do and would rather run this analysis on the CHPC.

Exemplary scientist that you are, you read up on some literature to determine what needs to be done in order to map reads to a genome. You discover the following tools:

* Bowtie2 - used for mapping reads to a genome
* Samtools 
* Bcftools

You know the CHPC has a plethora of tools available, but to be sure, you check first.


$ > ssh user@lengau.chpc.ac.za
$ > module add chpc/BIOMODULES
$ > module avail 2>&1 | grep -P "bowtie|samtools|bcftools"
bcftools/1.3.1
bowtie2/2.2.9
samtools/0.1.19
samtools/1.3
samtools/1.3.1

You’ll use bowtie2/2.2.9 and samtools/1.3.1. You make a note of this. These modules will need to be loaded before you can use either of the tools. You are unfamiliar with the tools that need to be used. After hours of consulting guides, you know the sequence of commands that will detect and report SNPs. Jotting down a sequence of commands is useless if you don’t understand the logical flow, so you note the procedure. Note that in a real-world scenario, the procedure may be a little different.

1. Obtain the reads reference genome
2. Quality trim the reads
3. Build a Bowtie2 index of the reference genome
4. Use Bowtie2 to map the *reads* to the reference
5. Convert the resultant SAM file to BAM format (using the reference)
6. Generate a "pile up" of the mapped reads
7. Call the SNPs

To simplify the exercise, I have already obtained the reference sequence, quality-trimmed the reads and built a bowtie2 index.
### Mapping

For the mapping procedure, you need to do the following:

1. Map using the generated bowtie2 index and the reads
2. Convert the SAM to BAM using the reference assembly and sort the BAM file

The commands:

> bowtie2 -p 24 -x k12_index_file -1 reads1 -2 reads 2 -S k12map.sam
> samtools view -but referencegenome.fasta k12map.sam > k12map.bam
> samtools sort k12map.bam k12map.sorted.bam

Explanation:

bowtie2 -p 24 -x k12 -1 reads1 -2 reads2 -S k12map.sam

We use bowtie2 to map the reads to the reference genome that is already a bowtie2 index file. Since the reads are paired-end, we need to specify this. The forward reads are specified with -1 and the reverse with -2. The output is specified with -S. To speed up the process, we can add the -p 24 command, which says we can use 24 threads, corresponding to 24 CPUs on a typical CHPC node.

samtools view -@ 24 -but k12.fasta k12map.sam > k12map.bam

This converts the SAM file into a more efficient BAM file. The -b switch indicates the output needs to be BAM formatted, the u indicates an uncompressed format and the t means a reference will be provided. Note that some tools allow you to combine separate parameters, in this case -u -b and -t into one. The output is to stdout so we redirect it to a file. Threads are specified differently for samtools view. So the -p flag is not ubiquitous across all tools.

samtools sort k12map.bam > k12map.sorted.bam

This sorts the mapped reads by the left-most coordinates. Some features require this step. The output is redirected to a file.

Being fancy

Of course, we could directly produce the final file, k12map.sorted.bam using pipes. Both bowtie and samtools allow output to be writted to stdout and the output can read from stdin. We can then chain these commands into a single line.

bowtie2 -p 24 -x k12 -1 reads1 -2 reads2 | samtools view -@ 24 -but ./EC1_scaffolds.fasta | samtools sort > k12map.sorted.bam

So only one output file will be written and this removes clutter from the working directory. If you’re not going to use “k12map.sam”, then why keep it?

Piling up and calling SNPs

The procedure to call the SNPs:

1. Run ``samtools mpileup`` to generate a ``pileup`` file
2. Run ``bcftools call`` to call the snps and generate a VCF file

> samtools mpileup -uf k12.fasta ./k12map.sorted.bam > k12map.sorted.mpileup
> bcftools call --output-type v --ploidy 1 -v -m -V indels k12map.sorted.mpileup > k12.vcf

Explanation:

samtools mpileup -uf k12.fasta ./k12map.sorted.bam > k12map.sorted.mpileup

Generate a pileup of the mapped reads, using a the K-12 genome as a reference and redirect output to k12map.sorted.mpileup.

bcftools call –output-type v –ploidy 1 -v -c -V indels k12map.sorted.mpileup > k12.vcf

We use bcftools to call the SNPs. The output is VCF specified by the --output-type v flag. We’re not really interested in insertion/deletion mutations, so we can ignore them with -V indels which skips those variants. We would also only like to report on positions that contain variants and speicify this with -v. The output is redirected to k12.vcf.

Building your recipe

So now that you know what to do, you can collate the commands into a recipe. We’ll use the “fancy” version of the mapping and the subsequent steps for SNP calling.

bowtie2 -p 24 -x k12 -1 reads1 -2 reads2 | samtools view -@ 24 -but ./EC1_scaffolds.fasta | samtools sort > k12map.sorted.bam samtools mpileup -uf k12.fasta ./k12map.sorted.bam > k12map.sorted.mpileup bcftools call –output-type v –ploidy 1 -c -v -V indels k12map.sorted.mpileup > k12.vcf

What is required of you, is to:

1. Run the tools as described above
2. BUT put the output in /home/wsmidt/lustre/chpc_lecture/[yourmachinename]

So you would need to include a folder creation command in your script:

mkdir -p /home/wsmidt/lustre/chpc_lecture/[yourmachinename]

The -p switch will allow the script to create a folder, while not moaning if it already exists. You can change to this directory with cd /home/wsmidt/lustre/chpc_lecture/[yourmachinename]. Running the commands in another folder is where things can get tricky. For instance, if you are in /home/wsmidt/lustre/chpc_lecture/[yourmachinename], pointing to the k12 bowtie2 index file as it is in the command would obviously fail. You therefore need to replace it with ../data/k12 which means:

  • ../ Go up one folder
  • data/ enter data
  • k12 use k12

The output with the redirection would be in the folder you are in. So the > k12map.sorted.bam would be in chpc_lecture/[yourmachinename] and not chpc_lecture/data.
